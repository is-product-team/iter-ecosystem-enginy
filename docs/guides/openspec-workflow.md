# Developer Guide: Working with OpenSpec & AI

This guide outlines the professional workflow for developing the Iter Ecosystem using **OpenSpec**, our AI-driven specification framework. By following this lifecycle, we ensure architectural consistency, automated documentation, and reliable feature implementation.

## 1. The Core Concept

**OpenSpec** is the "Source of Truth" for the entire project. It contains the technical specifications of our architecture, database, and business capabilities. 

When working with an AI assistant (like Antigravity), the specification acts as the shared mental model. **Code follows the spec, and the spec reflects the code.**

## 2. The Development Lifecycle (OPSX)

We use the **OPSX (OpenSpec eXchange)** workflow to manage every change in the system.

### Phase 1: Exploration (`/opsx-explore`)
Before implementing a complex feature or debugging a critical issue, use the exploration mode.
- **Goal**: Analyze the problem, check existing specs, and clarify requirements.
- **Outcome**: A clear understanding of the path forward without modifying code yet.

### Phase 2: Proposal (`/opsx-propose`)
Once the requirement is clear, generate a formal change proposal.
- **Action**: Run `/opsx-propose` describing the feature.
- **Outcome**: The AI creates a "Change Directory" with:
    - `design.md`: Technical design matching the system architecture.
    - `spec.md`: Updates to the relevant capability specifications.
    - `tasks.md`: A granular checklist for implementation.

### Phase 3: Implementation (`/opsx-apply`)
With an approved proposal, the implementation begins.
- **Action**: Run `/opsx-apply`.
- **Outcome**: The AI works through the `tasks.md` checklist, modifying code and verifying changes progressively.

### Phase 4: Archival (`/opsx-archive`)
After the feature is verified and pushed.
- **Action**: Run `/opsx-archive`.
- **Outcome**: The global specifications in `openspec/specs/` are updated with the new changes, and the temporary change directory is archived.

## 3. Best Practices for Developers

- **Spec-First Mentality**: If you change a core rule (e.g., maximum students in Modality C), update the spec in `openspec/specs/capabilities/scheduling.md` first.
- **Granular Commits**: Ensure each `/opsx-apply` task corresponds to a logical commit if possible.
- **Reviewing Proposals**: Always review the `design.md` generated by the AI before proceeding to implementation. It is easier to fix a design flaw than a code bug.
