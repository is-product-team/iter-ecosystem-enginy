# Proposal: Fix Docker Build Dependency Sync

## Goal
Resolve the `npm ci` failure occurring during Docker builds in the CI/CD pipeline.

## Problem
The current `Dockerfile` implements a `pruner` stage that runs `turbo prune` twice into the same output directory (`/app/out`).
1. `RUN turbo prune @iter/web --docker`
2. `RUN turbo prune api --docker`

The second command overwrites the `package-lock.json` and metadata generated by the first. When the `builder-web` stage tries to install dependencies, it uses a lockfile that only contains `api` dependencies, leading to "Missing: ... from lock file" errors.

## Solution
Split the `pruner` stage into two distinct stages: `pruner-web` and `pruner-api`. This ensures that each application builder receives a context and lockfile perfectly pruned for its own dependency tree.

## Risks & Considerations
- **Disk Space**: Running two prune stages might slightly increase the initial build time/cache size, but it is the correct way to handle multi-app monorepo builds in Docker.
- **Lockfile Synchronization**: The local `package-lock.json` must be up to date for `npm ci` to work in the container.
